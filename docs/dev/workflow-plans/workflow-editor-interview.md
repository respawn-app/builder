Known context:
- Existing product terminology defines workflows as durable directed graphs with nodes, edges, node groups, transition groups, graph revision, workflow drafts, validation contexts, project workflow links, task placements, and task board read models.
- Existing server service contract exposes workflow create/update/list/get, add/update node/group/transition group/edge, delete node group, project links, validation, tasks, comments, activity, board read models, and subscriptions. It does not expose full graph-definition CRUD deletes for nodes, transition groups, edges, or workflows.
- Existing GUI MVP PRD explicitly excludes visual workflow authoring; current GUI primarily exposes project home, workflow Kanban board, task detail, task creation, comments, attention, and runtime actions.
- Current desktop API schemas focus on board/task read models, not workflow definition graph editor read models.
- Desktop design constraints: server is authoritative; GUI is a remote-control surface; island-style UI; adaptive layouts; loading/error states; infinite scroll/pagination for unbounded collections; no hardcoded strings/colors/fonts.
- Smart architecture review completed and integrated. Key corrections: active project-workflow link validation must use `workflow.listProjectLinks`; board menu edit icon must avoid nested interactive controls; validation model must include transition group and related IDs; editor route must be added to edge-to-edge layout handling; license acceptance belongs in `docs/dev/decisions.md`.

Locked product decisions:
- v1 GUI scope: read-only graph-definition viewer. Backend/service/client contracts can be made CRUD-complete enough for v2, but v1 should not add workflow authoring forms or graph mutations.
- Runtime/task overlay: no runtime overlay in workflow editor. Runtime is already handled by Kanban board/task detail. The editor is the editor.
- Entry point: in the board view "+" edit/hover menu workflow list, add a small edit icon next to each workflow. This is the only v1 entry point.
- Graph revisions: ignore revision differences in v1 and show current graph without stale-revision warnings.
- Canvas density: MVP canvas shows node name and agent role as boxes. Edges show name/id as floating labels. Nothing else on the canvas for MVP. Use a library for infinite canvas/zoom/pan behavior.
- Canvas library: standardize on `@xyflow/react` / React Flow for v1 and future v2 editing, wrapped behind Builder-specific components.
- Layout positioning: v1 computes deterministic client-side auto-layout from graph structure; v2 can introduce persisted coordinates when visual editing exists.
- Layout orientation: left-to-right only in v1, matching Kanban/workflow progression.
- Viewer controls: zoom in/out, fit-to-view, and reset zoom. No minimap.
- Selection: v1 supports selection highlight only; no inspector/details panel.
- Destination shape: full board-replacing route/screen. Back/home/title are handled by existing app chrome/navigation.
- Rendered node set: render all graph nodes, including start/backlog, join, agent, and terminal/done, so topology is complete.
- Node role text: show blank role line when no subagent role exists.
- Edge label: label each edge with transition group display name if present, otherwise `transition_id`; for fan-out multi-edge groups append edge key only when needed to disambiguate.
- Node groups: render groups visually as boxes with boxes inside.
- Future CRUD boundary: v1 implements viewer only and records a concrete v2 CRUD gap list/contract plan. Do not expand backend mutation surface until editing UX is designed.
- Invalid/draft display: render the graph anyway and show the exact workflow issue component used by Board route for validation problems.
- Disconnected/isolated nodes: render every node, including disconnected nodes, placing isolated components below the main flow with a subtle warning/validation surface.
- Group layout semantics: treat groups as compound graph nodes that dictate layout hierarchy before edge layout. Parallel nodes should arrange vertically in a column.
- Gesture model: wheel zooms; drag empty canvas pans; node drag disabled in v1; buttons provide zoom/fit/reset.
- Parallel branch layout: in left-to-right flow, a source node at depth N can connect to a group edge or split into arrows into subnodes of a group at N+1; arrows merge into the next aggregating node at N+2.
- Layout engine: add/use ELK.js (`elkjs`) for deterministic graph layout with compound nodes/groups.
- Layout engine license: accept `elkjs` despite EPL-2.0 license because group/parallel layout quality matters for v1; document dependency/license acceptance in `docs/dev/decisions.md`.
- Initial viewport: fit the whole graph to view on first open; no persisted viewport in v1.
- Route identity: project-scoped route `/projects/:projectId/workflows/:workflowId/editor`.
- Board menu edit icon opens that workflow in the editor without changing board selection; returning to board preserves prior board state unless user separately selected a workflow.
- Validation mode: use `execution` validation for editor issue display.
- Live refresh: subscribe/refetch for current project/workflow using existing workflow project event stream pattern.
- Node group behavior: non-interactive visual containers with group label; no collapse/expand in v1.
- Edge visual style: curved edges.
- Editor controls placement: single top-left vertical floating island like Photoshop. Back/home/title are already handled by the existing chrome/navigation framework, so the canvas island is for graph controls rather than duplicating nav/title.
- Validation graph highlighting: subtle danger outline/tint on affected nodes/edges only; no extra labels or inspector.
- Board menu icon label/tooltip: "Edit workflow" even though v1 is read-only.
- Expected scale: optimize for 5-50 nodes and 5-100 edges; graceful behavior up to ~200 nodes; no special virtualization in v1.
- v2 CRUD architecture scope: graph-definition entities only: workflow metadata, node groups, nodes, transition groups, edges. Project links stay elsewhere; task/runtime CRUD stays in Kanban/task detail.
- Keyboard interactions: add basic shortcuts in v1: `+`/`-` zoom, `0` reset, `f` fit.
- Cycles/loops: render cycles as backward/looping curved edges; validation highlighting handles problematic loops.
- Node kind styling: use subtle theme-token accents/icons per kind while preserving the same sparse node box.
- Text overflow: fixed-size node boxes with 2-line clamped node names and 1-line ellipsized role; edge label chips ellipsize with full value in title/accessibility. If possible, allow visual node resizing.
- Canvas background: subtle dotted grid/background pattern using theme tokens.
- Read-only affordance: no explicit read-only indication; just omit editing controls.
- Node resize: allow ephemeral client-only resizing in v1 if practical; not persisted and resets on reload/refetch.
- Direct route with unlinked workflow: show an error/blocker that the workflow is not linked to the project; do not display unlinked workflows in project context.
- Viewport on live refetch: preserve current pan/zoom; show a small non-blocking "workflow updated" toast/status; fit only on first load or explicit Fit.
- Search/focus: no graph-aware search in v1; use browser find only.
- Empty groups: render empty group boxes with a placeholder label.
- Selection across refetch/layout: clear selection after refetch/layout.
- Chrome title: workflow name, with graph revision in a compact suffix if visually acceptable within the existing string-only chrome title.
- Group labels: show group display name; fall back to group key if needed. Empty groups show an "Empty group" placeholder inside.
- Fan-out/fan-in edge drawing: use whatever React Flow produces by default.
- Layout failure: show clear error state with retry and technical detail using the reusable error layout.

Architecture draft:
- Data source: use existing `workflow.get` for the graph definition and `workflow.validate` with `execution` mode for issue display. The desktop client needs new models/schemas for workflow definition entities because current desktop schemas are board/task-focused.
- Project scoping: because the route is `/projects/:projectId/workflows/:workflowId/editor`, validate that the workflow is actively linked to the project before rendering. Use `workflow.listProjectLinks(projectID)` as the authoritative active-link gate (`project_id` matches, `workflow_id` matches, `unlinked_at_unix_ms === 0`). Use `workflow.board.get(projectID, workflowID)` only for `latest_event_sequence` and other board context that already exists; do not rely on the board picker as the link authority.
- Route/navigation: add a TanStack route under the existing app chrome. Add `openWorkflowEditor(projectID, workflowID)` to `useAppNavigation`. Route persistence should continue to persist only project board routes unless product direction changes; editor reload/deep links are allowed but should not become the restored startup destination by default.
- Board entry point: extend `BoardHoverMenu` workflow rows with a separate small edit button. Current `WorkflowOption` composes an interactive `Item` row, so implementation must avoid nested buttons: refactor the workflow row into sibling controls or add a non-button composition mode before adding the edit icon. Row click keeps selecting the workflow. Edit button navigates to the editor for that workflow without changing current board selection.
- Shared validation issue UI: extract the Board route's `WorkflowValidationIssues` renderer into a shared board/workflow UI component so the editor uses the exact same issue list inside `FloatingNoticeIsland`. The editor issue island should be fed from `workflow.validate` execution-mode errors rather than board `valid_for_task_creation`.
- Subscriptions/freshness: reuse the existing workflow project event subscription pattern for the selected project/workflow, seeded from `workflow.board.get` `latest_event_sequence`. Existing workflow event queries include both project-specific events and global workflow-definition events (`project_id = ''`), so the editor should refetch when event `workflow_id` matches the open workflow or when a project-specific workflow-link event affects the open project/workflow. Preserve pan/zoom on refetch, clear selection, show a non-blocking updated status/toast, and do not refit unless first load or explicit Fit.
- Refetch event filter: refetch definition/layout/validation for `resource = "workflow"` and actions `node_added`, `node_updated`, `node_group_added`, `node_group_updated`, `node_group_deleted`, `transition_group_added`, `transition_group_updated`, `edge_added`, or `edge_updated` when `workflow_id` matches the open workflow. Refetch link gate and board sequence for `resource = "workflow_link"` and actions `linked`, `default_changed`, or `unlinked` when `project_id` matches the open project and either `workflow_id` matches the open workflow or `changed_ids` may affect the active link. Task/comment/run events are no-op for the editor except they may update global board surfaces elsewhere.
- Canvas boundary: wrap `@xyflow/react` behind Builder-owned workflow-editor components. Feature components should consume Builder view models, not raw server DTOs or layout-library types everywhere.
- Layout boundary: isolate ELK conversion in a pure layout module that maps `WorkflowDefinition` plus validation markers into positioned React Flow nodes/edges. This keeps EPL-2.0 `elkjs` usage localized and lets v2 swap/persist layout later.
- Group layout: represent node groups as React Flow parent/group nodes or equivalent compound visual nodes only. The data model has edges to nodes, not edges to groups; fan-out/fan-in remains transition-group edges to child node targets. Group nodes should contain child workflow nodes, show group display name or key, and render an empty placeholder for empty groups. ELK should receive group hierarchy before edge layout.
- Graph semantics: render all nodes. Agent nodes show name plus role; non-agent/no-role nodes have a blank role line. Edges derive source node from transition group and target from edge. Labels use transition display name or transition ID, with edge key appended only when needed to disambiguate fan-out.
- Visual semantics: use theme tokens only. Sparse node boxes get subtle kind accents/icons. Edge labels are floating chips. Curved edges are preferred; fan-out/fan-in uses React Flow defaults. Validation markers apply a subtle danger outline/tint to affected nodes/edges only.
- Interactions: node drag disabled; selection highlight only; selection clears after refetch/layout. Empty-canvas drag pans, wheel zooms, toolbar controls zoom/fit/reset, and shortcuts `+`, `-`, `0`, `f` map to the same actions. Ephemeral visual node resize is allowed if practical, but not persisted and resets on reload/refetch.
- States: use standard loading and reusable error layouts. If graph data loads but layout fails, show reusable `ErrorState` with retry and technical detail rather than rendering a misleading fallback graph.
- Dependencies: add `@xyflow/react` and `elkjs` to `@builder/desktop` dependencies and allowlist them in `apps/dependency-policy.json`. Current `@xyflow/react` package metadata reports MIT and React peer `>=17`; `elkjs` reports EPL-2.0 and has been explicitly accepted for v1. Record the license acceptance in `docs/dev/decisions.md`; dependency policy only records allowed package names unless its schema is deliberately extended.
- V2 contract notes: do not add missing delete/mutation RPCs in v1. Record v2 gaps: graph-definition CRUD should cover workflow metadata, node groups, nodes, transition groups, and edges; project links and task/runtime mutations remain outside the graph editor.
